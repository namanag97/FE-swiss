Last month we crossed 10 million API calls per day. Here's how we built the infrastructure to handle it — and what we learned along the way.

## Architecture Overview

Our system is split into three tiers: edge, application, and data. Each tier scales independently.

```
  System Architecture
  ================================

  Clients (Web, Mobile, API)
          |
          v
  +--------------------+
  |    Edge Layer       |    30+ global PoPs
  |    (CDN + Cache)    |    < 5ms cache hits
  +--------------------+
          |
          v
  +--------------------+
  |    App Layer        |    Auto-scaling pods
  |    (API Gateway)    |    Rate limiting
  |    (Auth + Routing) |    Request validation
  +--------------------+
          |
     +----+----+
     |         |
     v         v
  +------+  +-------+
  | Read |  | Write |       Read/write splitting
  | Pool |  | Pool  |       Connection pooling
  +------+  +-------+
     |         |
     v         v
  +--------------------+
  |    Data Layer       |    PostgreSQL (primary)
  |    (DB + Cache)     |    Redis (hot cache)
  |                     |    S3 (blob storage)
  +--------------------+
```

## The Read Path

90% of our traffic is reads. We optimized aggressively for this case.

```
  Read Request Flow (avg 12ms)
  ================================

  Client Request
       |
       v
  [Edge Cache] --hit--> Response (< 5ms)
       |
      miss
       |
       v
  [API Gateway]
       |
       v
  [Redis Cache] --hit--> Response (< 15ms)
       |
      miss
       |
       v
  [Read Replica]
       |
       v
  [Cache Write] --> Response (< 50ms)
```

Key decisions:
- **Edge caching** for static and semi-static data (user profiles, org settings)
- **Redis** for hot data with 5-minute TTLs
- **Read replicas** in every region to keep queries close to users
- **Connection pooling** with PgBouncer — we never open a fresh connection per request

## The Write Path

Writes are more complex. We use an event-driven architecture to keep writes fast and consistent.

```
  Write Request Flow
  ================================

  Client Request
       |
       v
  [API Gateway]
       |
       v
  [Validation + Auth]
       |
       v
  [Primary DB] --> ACK to client (< 30ms)
       |
       v
  [Event Bus]
       |
       +--------+--------+--------+
       |        |        |        |
       v        v        v        v
   [Search]  [Cache]  [Analytics] [Webhooks]
   reindex   invalidate  aggregate  fan-out
```

The key insight: **acknowledge the write immediately**, then process side effects asynchronously. The user sees sub-30ms response times, while reindexing, cache invalidation, and webhook delivery happen in the background.

We alert on any of these crossing thresholds. Cache hit rate dropping below 90% is a red flag — it usually means a cache invalidation bug.

## Lessons Learned

1. **Profile before you optimize.** We wasted two weeks optimizing a serialization path that was only 3% of total latency.

2. **Connection pooling matters more than query optimization.** Cutting connection overhead from 15ms to 0.5ms was our single biggest win.

3. **Cache invalidation is the hardest problem.** We eventually settled on event-driven invalidation with a 5-minute max TTL as a safety net.

4. **Read replicas need monitoring.** Replication lag caused subtle bugs in our dashboard — users would create something and not see it for 2-3 seconds.
